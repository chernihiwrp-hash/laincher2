# ZYNX — ПОЛНАЯ ИНСТРУКЦИЯ ПО НАСТРОЙКЕ

Система из 3 частей:
1. **Клиент** (Python → .exe) — то, что запускает игрок. Просит ключ.
2. **Сервер** (Vercel + Supabase) — проверяет ключи и сроки.
3. **Админка** (admin.html) — где ты выдаёшь ключи.

```
Игрок вводит ключ в .exe  ─►  твой Vercel /api/license  ─►  Supabase (база ключей)  ─►  можно/нельзя
Ты выдаёшь ключи         ─►  admin.html  ─►  твой Vercel /api/admin-keys  ─►  Supabase
```

================================================================================
## A. ЧТО ЛЕЖИТ ГДЕ (структура)
================================================================================

### 1) РЕПОЗИТОРИЙ КЛИЕНТА (Python) — у тебя на компе
Здесь собираешь .exe. В репозиторий/папку:
```
client/
  Launcher.py          ← основной код (лаунчер, оверлеи, HUD)
  zmenu_section.py     ← внутриигровое меню (клавиша 0)
  license_gate.py      ← гейт ключа  (ТУТ вписываешь адрес сервера)
  build_merge.py       ← склейка всего в main.py
  → после сборки появляется main.py  (его и упаковываешь в .exe)
```
Папки со звуками НЕ кладёшь в репозиторий — они создаются рядом с .exe сами:
`REDUXSOUNDS/`, `interface sounds/` (click.wav, toggle.wav, slider.wav), `moan/`.

### 2) РЕПОЗИТОРИЙ/ПРОЕКТ VERCEL — на GitHub → Vercel
```
vercel-project/
  api/
    license.ts         ← (это файл api_license.ts, переименуй)
    admin-keys.ts      ← (это файл api_admin_keys.ts, переименуй)
  public/
    admin.html         ← веб-панель ключей
  package.json         ← с зависимостью @supabase/supabase-js
```
Если у тебя УЖЕ есть Vercel-проект (сайт RP) — просто добавь туда `api/license.ts`,
`api/admin-keys.ts` и `public/admin.html`. Отдельный проект не обязателен.

### 3) SUPABASE — в облаке
SQL-скрипт `keys_schema.sql` (один раз выполнить). Создаёт таблицы
`license_keys` и `license_checks`.

================================================================================
## B. SUPABASE (база) — 5 минут
================================================================================
1. Открой свой проект Supabase → **SQL Editor** → New query.
2. Вставь содержимое `keys_schema.sql` → **Run**.
3. Возьми два значения (Settings → API):
   - **Project URL**           → это `SUPABASE_URL`
   - **service_role secret**   → это `SUPABASE_SERVICE_ROLE_KEY` (СЕКРЕТ! только на сервер)

================================================================================
## C. VERCEL (сервер) — 10 минут
================================================================================
1. В проект Vercel положи файлы (см. структуру выше):
   - `api/license.ts`     (из `api_license.ts`)
   - `api/admin-keys.ts`  (из `api_admin_keys.ts`)
   - `public/admin.html`
2. `package.json` должен содержать зависимость:
```json
{ "dependencies": { "@supabase/supabase-js": "^2.45.0" } }
```
   (если ставишь в существующий проект — просто `npm i @supabase/supabase-js`)
3. Vercel → проект → **Settings → Environment Variables** добавь 3 штуки:
```
SUPABASE_URL              = https://xxxx.supabase.co
SUPABASE_SERVICE_ROLE_KEY = <service_role secret из Supabase>
ADMIN_TOKEN               = <придумай длинный пароль, напр. zynx_9f3a7c...>
```
4. Деплой (git push или кнопка Redeploy).
5. Проверь, что открывается:
   - `https://твой-проект.vercel.app/admin.html`  (панель)
   - endpoint для клиента: `https://твой-проект.vercel.app/api/license`

================================================================================
## D. PYTHON-КЛИЕНТ — что вписать
================================================================================
Открой **`license_gate.py`**, найди строку и впиши свой адрес:
```python
LICENSE_API = "https://твой-проект.vercel.app/api/license"
```
Больше в коде менять ничего не нужно.

================================================================================
## E. СБОРКА .EXE
================================================================================
1. Поставь зависимости (один раз):
```
pip install pyinstaller PyQt6 pywin32 pygame keyboard winsdk pycaw comtypes requests
```
2. Склей всё в один main.py:
```
python build_merge.py
```
   (создастся `main.py` — это весь клиент целиком, с гейтом ключа)
3. Собери .exe:
```
pyinstaller --onefile --noconsole --name ZYNX --collect-all winsdk --collect-submodules pygame --hidden-import win32api --hidden-import win32gui --hidden-import win32process main.py
```
4. Готово: `dist\ZYNX.exe`. Его и раздаёшь.

### Защита от взлома (по желанию, по нарастанию):
- Обфускация: `pip install pyarmor` → `pyarmor gen --pack onefile main.py`
- Поверх .exe — VMProtect/Themida (максимум).
- Главное уже сделано: срок/валидность решает СЕРВЕР, не клиент.

================================================================================
## F. КАК ВЫДАВАТЬ КЛЮЧИ
================================================================================
Вариант 1 — **веб-панель** (удобно):
  открой `admin.html` → впиши адрес API и `ADMIN_TOKEN` → «Создать ключ»
  (срок: 30д / 1–12 мес / бессрочно). Ключ копируется автоматически.

Вариант 2 — **скрипт** `keygen.py` (впиши там SUPABASE_URL и SERVICE_ROLE_KEY):
```
python keygen.py --months 1     # на месяц
python keygen.py --months 12    # на год
python keygen.py --forever
```

================================================================================
## G. ПРОВЕРКА, ЧТО ВСЁ РАБОТАЕТ
================================================================================
1. Создай тестовый ключ в панели.
2. Запусти ZYNX.exe → введи ключ → должно пустить.
3. В панели отключи ключ (кнопка «откл») → перезапусти .exe → не пустит.
4. Создай ключ на 0 дней нельзя — для теста сделай на 1 день и проверь позже,
   либо вручную в Supabase поставь `expires_at` в прошлое → клиент скажет «истёк».

ГОТОВО. Теперь у тебя полноценная система ключей со сроками и веб-панелью.
