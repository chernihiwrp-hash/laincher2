# ZYNX — сборка в один .EXE + система лицензий

## ЧАСТЬ 1. Один .EXE (PyInstaller)

1. Установи зависимости (один раз):
```
pip install pyinstaller PyQt6 pywin32 pygame keyboard winsdk pycaw comtypes
```

2. Собери `launcher.py` (это уже собранный main с гейтом) в один файл:
```
pyinstaller --onefile --noconsole --name ZYNX ^
  --hidden-import win32api --hidden-import win32gui --hidden-import win32process ^
  --hidden-import winsdk.windows.media.control ^
  --hidden-import winsdk.windows.storage.streams ^
  --collect-all winsdk ^
  --collect-submodules pygame ^
  launcher.py
```
(в одну строку, без `^`, если не cmd)

3. Готовый файл: `dist\ZYNX.exe`.

### Важно про папки рядом с .exe
Программа сама создаёт рядом с .exe папки:
- `REDUXSOUNDS` — звуки выстрелов (катомные);
- `interface sounds` — `click.wav`, `toggle.wav`, `slider.wav`;
- `moan` — (если используешь).
Положи туда нужные звуки **после** первого запуска (или создай их сам рядом).

### Если антивирус ругается
PyInstaller-сборки часто ловят ложные срабатывания. Варианты: подписать .exe сертификатом, добавить в исключения, либо собрать `--onedir` (папкой) вместо `--onefile`.

## ЧАСТЬ 2. Система ключей (срок: месяц/год + привязка к ПК)

Как это работает: при запуске .exe спрашивает ключ → отправляет его + ID компьютера на **твой сервер** → сервер смотрит в базе срок и привязку → разрешает или нет. **Всё решает сервер**, не клиент.

### Шаг 1 — База (Supabase)
В Supabase → SQL Editor выполни `server/keys_schema.sql`. Появятся таблицы `license_keys` и `license_checks`. RLS включён — анонимно базу не читать.

### Шаг 2 — Эндпоинт (Vercel)
Положи `server/api_license.ts` в свой Vercel-проект как `api/license.ts`.
Добавь переменные окружения в Vercel:
- `SUPABASE_URL`
- `SUPABASE_SERVICE_ROLE_KEY` (секретный ключ Supabase — **только на сервере**)
Задеплой. Эндпоинт будет: `https://твой-проект.vercel.app/api/license`.

### Шаг 3 — Вписать адрес в клиент
В `license_gate.py` строка:
```python
LICENSE_API = "https://ВАШ-ПРОЕКТ.vercel.app/api/license"
```
замени на свой адрес, пересобери (`python build_merge.py`) и заново сделай .exe.

### Шаг 4 — Выдавать ключи
`server/keygen.py` — впиши `SUPABASE_URL` и `SERVICE_ROLE_KEY`, затем:
```
python keygen.py --days 30   --note "ник"     # на месяц
python keygen.py --months 12 --note "ник"     # на год
python keygen.py --forever                    # бессрочный
```
Скрипт выдаст ключ вида `A1B2-C3D4-E5F6-G7H8` — отдаёшь покупателю.

Управление вручную (в Supabase):
- отключить ключ: `update license_keys set active=false where key='...';`
- сбросить привязку к ПК (сменил комп): `update license_keys set hwid=null where key='...';`

## ЧАСТЬ 3. Честно про «защиту от взлома»

То, что **реально защищает** (сделано):
- **Срок и валидность ключа проверяет сервер**, а не клиент — нельзя «перевести часы» на ПК и продлить.
- **Привязка к ПК (HWID)** — один ключ = один компьютер, шарить бесполезно.
- **service_role-ключ только на сервере** — базу ключей со стороны не прочитать.
- Оффлайн-грейс всего 3 дня, потом обязателен онлайн-чек.

Чего Python (даже в .exe) **не может** гарантировать:
- .exe из PyInstaller **распаковывается и декомпилируется** — опытный человек может вырезать сам вызов проверки и собрать «пиратку».
- Полностью это лечится только если **важная логика выполняется на сервере** (например, сервер отдаёт что-то нужное приложению только при валидном ключе), либо коммерческим протектором (VMProtect/Themida + обфускация).

Вывод: эта система **отсекает 95% обычных пользователей** (шаринг ключей, просроченные, «накрутка времени»), но не остановит профессионального реверс-инженера. Для твоей аудитории (Roblox-комьюнити) этого более чем достаточно. Хочешь жёстче — выноси часть функций (например, генерацию ClientAppSettings/пресеты) на сервер и отдавай только по валидному ключу: тогда «пиратка» будет нерабочей без сервера.
